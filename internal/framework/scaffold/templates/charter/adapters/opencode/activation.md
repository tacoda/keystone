# opencode — Activation

How ambient corpus content gets loaded into opencode, where the menu
lives, and how context resets work.

## The menu

opencode reads `AGENTS.md` at the repo root on session start. Keystone
drops one there pointing at `.charter/README.md` and the index.

## Where runtime config lives

- `AGENTS.md` at repo root — project menu pointing at the charter (written
  by the installer).
- `opencode.json` at repo root — project config: model selection, the
  `mcp` server map, and other opencode settings. The installer only
  touches the `mcp` key (via `keystone mcp install --agent opencode`);
  everything else is user-owned.
- `.opencode/` — projected primitives: `skills/`, `agents/`, `commands/`,
  and `rules/` (guide rule-shims). Regenerated by `keystone project` from
  `.charter/`. Do not hand-edit; edits are overwritten and the drift sensor
  flags them.

## Ambient loading

`AGENTS.md` is read at session start. The full corpus (`.charter/`)
is read on demand inside the conversation — not autoloaded.

- `principles/` is referenced from `AGENTS.md` so the agent knows where to
  find it.
- `idioms/<stack>/` loads when the **orient** action detects the touched
  region.
- `domain/` loads at the start of any task.
- `state/` loads on demand from **orient**.
- `process/<phase>.md` loads when the agent enters that phase.

## Projection opt-in

`.opencode/` projection is opt-in. Add opencode to the `adapters` list in
`keystone.json`:

```json
{ "adapters": ["opencode"] }
```

Then run `keystone project` (or `keystone watch` for continuous
re-projection). Without the opt-in, opencode still consumes the charter by
reading `.charter/INDEX.json` and the referenced bodies directly — the
projection just makes skills/subagents/commands discoverable as
first-class opencode primitives.

## Context-reset primitive

To reset context after a flywheel write (**synthesize** or **audit**),
start a new opencode session. The next session reads the freshly written
corpus.

## Rules

opencode loads rules through the `instructions` array in `opencode.json`
(combined with `AGENTS.md`). `keystone project` projects globbed idiom
guides to `.opencode/rules/*.md` and registers that glob in `instructions`
automatically — so the rule-shims load every session.

Caveat: opencode's `instructions` are **always-on**. Unlike Cursor or
Claude Code, opencode does not gate rules per edited file, so every
projected guide loads on every turn regardless of which files you touch.
For a large idiom set this is heavier than the lazy-by-region ideal.

## Lazy-by-region — manual

For per-region loading beyond the always-on `instructions` set, the
**orient** action implements lazy-by-region manually: the agent reads the
touched paths, walks `.charter/corpus/state/CODEBASE_STATE.md` to map paths
to stacks, consults `.charter/corpus/state/GLOBS_INDEX.md` to gate per-guide
loading on declared `globs:`, and reads the resulting guides on demand.
opencode also supports lazy `@file` references in `AGENTS.md` for the same
load-on-need pattern.
